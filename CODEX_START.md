## 2. 项目目标

BidRadar 面向纸箱、包装箱和纸制品包装业务，将公开招标、生产项目、环评、扩产及企业采购入口等线索整理为可跟进的结构化数据。

核心业务链路包括：

- 采集公开招标和项目线索。
- 进行包装行业相关性过滤、地区识别、价值评分和推荐。
- 合并历史结果并形成正式 Excel/TXT 输出。
- 从主结果提取高价值线索，维护目标企业、客户池和跟进任务。
- 发现、排序、导入和验证企业采购入口。
- 搜索联系方式候选，并在人工确认后导入客户池。

## 3. 当前架构

### 分层结构

- **采集层：** `crawler.py`、企业入口脚本，以及生产、环评、扩产项目监控脚本负责获取公开信息。
- **业务层：** `industry_filter.py`、`scoring.py`、客户池、跟进、候选导入等模块执行既有业务规则。
- **数据层：** `paths.py` 以项目根目录为基准统一登记正式 Excel/TXT 路径；`utils/excel_helper.py` 提供安全 Excel 读写。
- **输出层：** 生成主结果、项目结果、客户池、状态表、候选表、调试 Excel 和 UTF-8 文本报告。
- **验证层：** `tests/` 覆盖离线回归、数据契约、安全写入、故障注入和跨工作目录路径场景。

### 主数据流

```text
公开网页 / 手工表 / 剑鱼 raw / 企业候选
    -> 采集或导入
    -> 行业过滤与地区识别
    -> 评分和推荐
    -> Excel 结果
    -> 高价值线索
    -> 客户池
    -> 跟进任务
```

### 主程序边界

`main.py` 依次调用采集、过滤、评分、Excel 导出和日报生成模块。当前 `USE_SAMPLE_DATA = False`，直接运行会访问公开网页，并可能更新正式 Excel/TXT；未经明确授权不得运行。

## 7. 下一步开发计划

1. 保持 V6.6.0 稳定，仅处理能够复现且影响安全或稳定性的缺陷。
2. 如需真实采集验收，先单独制定并确认联网授权、访问频率、隔离输出、正式数据备份和回滚方案。
3. 在隔离输出中验证外部网站结构、访问限制、登录状态和采集兼容性，避免直接写入正式结果。
4. 受控验证 Excel 被占用、目录不可写及多文件失败后的人工恢复流程。
5. 后续版本再评估依赖锁定、结构化日志和更强的多文件事务能力。
6. 任何业务规则、评分、过滤、唯一 ID、字段或数据流调整均应另立版本，不得混入 V6.6.0 稳定维护。

## 8. Codex 工作规则

1. 开始工作前阅读 `CODEX_START.md`、`README.md`、`WORK_LOG.md`、`FILE_MAP.md`、`docs/PROJECT_STATUS.md` 和 `docs/ARCHITECTURE.md`；涉及字段时再阅读 `docs/EXCEL_CONTRACTS.md`。
2. 修改前检查 Git 状态，保护用户已有改动；未经明确要求不得提交或 push。
3. 默认进行只读检查。未经明确授权，不运行 `main.py`、`run_bidrader.bat`、真实采集、浏览器自动化、URL 验证或其他可能联网的脚本。
4. 未经明确授权，不运行任何会读取、创建或写回正式 Excel/TXT 的脚本；不得以正式模式代替离线验证。
5. 不直接修改、覆盖、移动或删除业务 Excel/TXT，也不修改 `backup/auto/` 中用于恢复的最新备份。
6. V6.6.0 稳定期不得修改既有业务逻辑，包括行业过滤、地区识别、评分、推荐、唯一 ID、历史合并、字段结构和人工字段保留规则。
7. 修改数据层前先确认 `paths.py` 登记、Excel 数据契约、涉及模块、读写文件、备份策略、失败行为和回滚方案。
8. 运行任何验证前明确网络边界和输出位置；优先使用离线、无字节码、隔离输出的验证方式，并确认正式数据不会变化。
9. 提交前检查不得包含 `*.xlsx`、`*.txt`、自动备份、临时文件、`.pyc`、`__pycache__` 或敏感信息。
10. 发现文件损坏、读取失败、外部修改、Excel 占用、意外数据变更或未授权联网行为时立即停止，并报告受影响文件和可用备份路径。
11. 完成工作后报告修改文件、验证范围、Git 状态和未验证事项；重要状态变化应同步更新项目文档。
